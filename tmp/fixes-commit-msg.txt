fix(release): package.json metadata + cross-platform build copy

Corrige les 2 failures du run v0.2.0 (run id 27758140677) — macOS ✅,
Windows ❌ + Linux ❌. Cause racine double identifiée via `gh api` logs.

### 🐛 Cause #1 — bash `cp -r` cassait sur windows-latest

`bun run build` invoque `cp -r` dans un shell dont la sémantique
diffère de GNU coreutils sur les runners Windows (msys/Git-Bash vs
PowerShell cp alias). Erreur exacte : `cp: illegal option -- r`.

**Fix** : nouveau script `scripts/copy-standalone-assets.mjs` qui utilise
`node:fs/promises` `cp(src, dest, { recursive: true, errorMode: "throw" })`.
- Cross-platform garanti par stdlib Node (≥ 16.7 / Bun ≥ 1)
- `errorMode: "throw"` au lieu de `"warn"` par défaut → les EACCES/ENOENT
  ne passent plus sous silence et plantent le job proprement
- `existsSync` pre-check avant copy → message d'erreur explicite
  ("source missing: did 'next build' complete?") au lieu d'une trace
  ENOENT indigeste

### 🐛 Cause #2 — .deb/.rpm exigeaient `author.email`/`description`/`desktopName`

electron-builder a planté Linux avec :
`Please specify author 'email' in the application package.json`
Debian contrôle fields (Author, Description) sont obligatoires pour les
`.deb`, similaires pour `.rpm`.

**Fix** : 4 nouveaux champs top-level dans `package.json` :
- `description` : tagline du jeu pour les metadata des paquets
- `author` : **objet** `{ name, email }` (string form silently ignored
  par electron-builder pour les .deb)
- `homepage` : URL GitHub
- `desktopName` : "Eldoria" = `productName` d'electron-builder.yml

### 🛡 Validation

- `package.json` reste JSON-valide (python json.load OK)
- `node --check scripts/copy-standalone-assets.mjs` = SYNTAX_OK
- Build script = `"next build && node scripts/copy-standalone-assets.mjs"`
  (plus aucun `cp -r` residue)

### ▶️ Suivant

Tag `v0.2.1` à pousser → release.yml re-fonctionne sur les 3 OS natifs.
v0.2.0 reste comme tag orphelin (release jamais publiée car matrix cassée).
